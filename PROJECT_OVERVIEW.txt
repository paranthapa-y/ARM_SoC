WHAT THIS REPOSITORY IS
--------------------------------------------------------------------------------
This is ARM's "Cortex-M3 DesignStart Eval" bundle (release CM3DesignStart-r0p0-
02rel0, 2017), used as the starting point for an in-house ARM Cortex-M3 SoC.
It is NOT a from-scratch SoC design — it is ARM's reference evaluation kit:
RTL for the SoC "uncore" (bus fabric, peripherals, memories) plus a licensed,
OBFUSCATED/encrypted model of the Cortex-M3 CPU core itself (ARM does not give
eval-kit users readable CPU RTL). Around this, the repo layers:
  - A cycle-based Verilog simulation testbench (run in ModelSim/Questa, VCS,
    or Xcelium) used to execute compiled firmware images against the SoC model.
  - A software tree (CMSIS device files, startup code, retarget/UART stdio,
    and ~20 test programs) that is compiled and loaded into that simulation.
  - A separate "boards/" bundle of prebuilt binaries for the physical MPS2+
    FPGA prototyping board (real hardware, not simulation).
  - Several optional/additional ARM IP bundles (m3designstart_iot, rtc_pl031,
    smm, trng) shipped alongside the core kit — see CONNECTIVITY section below
    for which of these are actually wired into the simulated system today.

In short: "SoC" here = Cortex-M3 core (opaque model) + CMSDK bus/peripheral
fabric + MPS2 FPGA-shield peripheral wrapper, all instantiated under a single
testbench top (tb_fpga_shield.v) that a Makefile-driven flow compiles and
simulates against one test program at a time.

blocks above are all on the search path but never actually instantiated.

END-TO-END FLOW: FROM `make run TESTNAME=hello` TO SIMULATED EXECUTION
--------------------------------------------------------------------------------
1. COMPILE THE FIRMWARE
   cd m3designstart/logical/testbench/testcodes/hello && make
   -> invokes testcodes/hello/makefile, target all_gcc (TOOL_CHAIN=gcc default):
      arm-none-eabi-gcc compiles startup_CM3DS.s + hello.c + retarget.c +
      uart_stdout.c(+ uart_stdout1.c, WIP) + system_CM3DS.c, links with
      software/common/scripts/cm3ds.ld, producing hello.o (ELF), then:
        objcopy -O binary  -> hello.bin   (raw memory image)
        objcopy -O verilog -> hello.hex   (verilog hex, alt. load path)
        objdump -S          -> hello.lst  (disassembly listing)
   Linker script memory map (cm3ds.ld):
        FLASH (code) : 0x00000000, 64 KB
        RAM  (data)  : 0x20000000, 63 KB
   This matches the ARMv7-M/Cortex-M3 boot convention: address 0 holds the
   vector table; word[0] = initial stack pointer (SP), word[1] = reset vector
   (address of Reset_Handler). This is how the CPU "knows where to start" —
   see your follow-up question, happy to go deeper on this.

2. COMPILE THE TESTBENCH
   cd execution_tb && make compile SIMULATOR=mti
   -> vlib/vlog compiles tb_fpga_shield.v and everything it pulls in via the
      tbench.vc file list (RTL + the obfuscated Cortex-M3 model), producing a
      ModelSim "work" library. Default simulator is mti (Questa/ModelSim),
      default toolchain gcc, DSM=no (i.e. using the obfuscated RTL core model,
      not the Carbon/DSM cycle model — DSM=yes would additionally enable
      register-visibility + TARMAC instruction tracing).

3. RUN
   make run TESTNAME=hello
   -> before invoking the simulator, the Makefile:
        - copies a CXDT.bin discovery image in (CXDT = debug/config image
          used by the CPU model's boot/discovery mechanism)
        - converts hello.bin into flash_main.ini via
              od -v -A n -t x8 --width=16 hello.bin | awk '{print $2$1}'
          i.e. a 16-byte-per-line hex dump, byte-order-swapped per word —
          this is the format the CPU model's internal flash reads on startup
          to preload code memory (so hello.bin ends up resident at address 0
          before the simulated core comes out of reset).
        - runs vsim in batch mode (`-c -do "run <time>; quit -f"`), piping
          the console output through `tee mti_hello_run.log`.
   -> Inside simulation, the CPU boots from address 0, jumps to Reset_Handler,
      SystemInit()/system_CM3DS.c configures the clock, then main() runs.
      hello.c calls UartStdOutInit() and prints via retarget.c's _write() ->
      UartPutc() -> CMSDK UART0 DATA register. The testbench module
      cmsdk_uart_capture_ard.v sniffs the UART TX line bit-by-bit and prints
      captured characters/lines to the simulation console/log with a
      timestamp. Sending byte 0x04 (EOT) on the UART, done in
      UartEndSimulation(), is the signal cmsdk_uart_capture_ard.v uses to end
      the simulation ($stop).

4. PASS/FAIL DETERMINATION
   `make runall` (and effectively the intent behind `make run`) determines
   PASS/FAIL by grepping the run log for the literal string:
       ** TEST PASSED **
   (execution_tb/Makefile line ~376). The ORIGINAL hello.c printed exactly
   this string after "Hello world". See FLAGS below — this string is
   currently commented out in the working tree, which breaks this check.
